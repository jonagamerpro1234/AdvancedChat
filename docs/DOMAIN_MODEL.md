# AdvancedChat Domain Model

> **Status:** Draft
>
> **Version:** 2.0.0-alpha.1
>
> **Last Updated:** 01-07-2026

## Overview

This document defines the domain model of AdvancedChat.

The purpose of this document is to describe the business concepts of the project before implementing them in code.

This document should answer one question:

> **What are we building?**

It does **not** describe implementation details, platform APIs, storage systems, or external integrations.

---

## Purpose

The domain model serves as the foundation of the entire project.

Every module, service, API, and implementation must be designed around the concepts defined in this document.

If a new concept is introduced into the project, it should first be documented here before implementation.

---

## Out of Scope

This document does **not** describe:

- Technical implementations.
- Storage engines.
- Platform-specific APIs.
- External integrations.
- Network architecture.
- Internal implementation details.

---

## Design Principles

- The domain comes before the implementation.
- Every concept must have a clear responsibility.
- The Core module must only contain business logic.
- Platform-specific code belongs outside the Core.
- New concepts should be introduced only when necessary.
- The implementation follows the domain.
- The domain never follows the implementation.

---

# Domain Concepts

Represents a player interacting with the communication system.

### Responsibilities

- Send messages.
- Receive messages.
- Belong to channels.
- Own metadata.

---

## Domain Rules

This section defines the fundamental rules of the domain.

Initial rules:

- Every message has exactly one sender.
- A player may belong to multiple channels.
- Every channel has a unique identifier.
- Formatting must never modify the original message.
- Services should communicate through interfaces whenever possible.

Additional rules will be added as the project evolves.
